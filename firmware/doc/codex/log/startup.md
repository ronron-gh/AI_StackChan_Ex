**進め方の提案**

1. 変更前に「目的・影響範囲・戻し方」を短く決める  
   例えば「長期記憶を追加したい」なら、いきなり `ChatGPT.cpp` に足すのではなく、まず「保存先」「LLMBase との責務分担」「Web UI/API の要否」「Module LLM / Realtime API への影響」を確認します。

2. 既存の境界を崩さない  
   アプリ的な機能は基本 `src/mod`、LLM 連携は `src/llm`、音声やハード依存は `src/driver`、設定は `StackchanExConfig` 側、という分離を守るのがよさそうです。特に `main.cpp` は肥大化しやすいので、登録・起動・イベント配送以上のロジックを増やしすぎない方針が良いです。

3. 小さい単位で実装する  
   「保存機能」「設定項目」「Web API」「画面操作」「LLM への注入」を一度に混ぜると後で読めなくなります。1ステップずつ、ビルド可能な状態を保ちながら入れるのが安全です。

4. 仕様メモをコードと一緒に更新する  
   既に [doc/fw_design.md](c:/git/AI_StackChan_Ex/firmware/doc/fw_design.md) があるので、機能追加のたびに「何のための機能か」「どのクラスが責務を持つか」「設定ファイル/API/タスクへの影響」を短く残す運用が良いです。巨大な設計書ではなく、後から自分が迷わないための索引にします。

5. 文字コードを壊さない  
   README は UTF-8 で読めましたが、`src` 配下の一部コメントは PowerShell 上で文字化けして見えています。今後編集するときは、ファイルごとの文字コードを確認して、不要な全体再保存やフォーマットを避けます。これはかなり大事です。

6. 機能ごとに「保守用の名前」を付ける  
   例えば「MemoryStore」「FunctionCallRegistry」「RealtimeAudioSession」みたいに、役割が後で読める名前を付けます。つぎはぎ感は、処理の場所よりも「なぜそこにあるか」が見えない時に出やすいです。

7. ビルド確認を標準手順にする  
   PlatformIO の環境が複数あります。少なくとも変更対象に応じて、例えば `m5stack-core2-realtime`、必要なら `m5stack-cores3-realtime` / `m5stack-atoms3r-realtime` も確認する、という基準を決めたいです。
