# AGENTS.md

このファイルは、AI_StackChan_Ex の firmware 配下を継続的に保守するための作業方針です。
Codex などのコーディングエージェントは、変更作業の前提としてこの内容を確認してください。

## 基本方針

- 既存の設計意図を優先し、場当たり的な追加を避ける。
- 変更前に、目的、影響範囲、既存機能への副作用、戻し方を短く整理する。
- 実装は小さい単位に分け、各段階でビルド可能な状態を保つ。
- 変更の理由が後から分かるように、必要に応じて `doc/fw_design.md` へ設計メモを残す。
- ユーザーが作業中の変更を勝手に戻さない。関係ない差分は触らない。

## 責務分担

- `src/mod`: ユーザーアプリケーション、モード切り替えで動作する機能。
- `src/llm`: LLM、Realtime API、Function Calling、会話履歴などの AI 連携。
- `src/stt`: 音声認識。
- `src/tts`: 音声合成。
- `src/driver`: ハードウェア、音声入出力、カメラ、Wake Word、Tap Detect など。
- `src/share`: 共通ユーティリティ。
- `src/WebAPI.*`: Web API と Web UI 連携。
- `src/StackchanExConfig.*`: YAML 設定と設定構造。
- `incbin`: Web UI の HTML/JS 埋め込み元。
- `doc`: 設計メモ、利用方法、保守用ドキュメント。

新機能は、まず上記のどこに責務を置くべきかを判断してから実装する。
`main.cpp` には、起動、初期化、タスク登録、イベント配送を中心に置き、機能固有ロジックを増やしすぎない。

## 実装ルール

- 既存クラス、命名、ifdef、PlatformIO 環境の使い方に合わせる。
- Core2、CoreS3、AtomS3R、Realtime API、Module LLM などのビルド条件に注意する。
- ハードウェア依存やサービス依存の処理は、可能な範囲で境界を分ける。
- メモリ使用量、タスクスタック、PSRAM、SPIFFS/SD、音声入出力の排他制御に注意する。
- API キー、Wi-Fi パスワード、個人情報をコードやドキュメントに追加しない。
- 日本語コメントや既存ドキュメントの文字コードを壊さない。不要な全体再フォーマットは避ける。
- コメントは「なぜそうするか」が必要な場所にだけ追加する。

## 変更手順

1. 関連ファイルを読み、既存の流れを把握する。
2. 変更方針と影響範囲を短く確認する。
3. 実装前に、作業単位のステアリングファイルを `doc/codex/steering/YYYYMMDD-short-topic.md` として作成する。
4. ステアリングファイルの内容をユーザーに提示し、レビューと了承を得てから実装に進む。
5. 最小限のファイルに実装する。
6. 必要なら `doc/fw_design.md` へ設計メモを追記する。
7. 可能な範囲で PlatformIO ビルドや静的確認を行う。
8. 最後に、変更内容、確認結果、残リスクを簡潔に報告する。

## ステアリングファイル

実装に入る前に、今回の作業方針を `doc/codex/steering/` に Markdown で残す。

- ファイル名は `YYYYMMDD-short-topic.md` とする。
- 目的、対象範囲、主な変更ファイル、実装方針、設計上の注意点、確認方法を簡潔に書く。
- ステアリングファイルを作成または更新したら、その内容をユーザーに提示し、レビュー完了の指示を受けてから実装を開始する。
- 作業中に方針が変わった場合は、必要に応じて同じファイルを更新する。
- ステアリングファイルは、その作業のコミットに含める。

## ビルド確認の目安

変更内容に応じて、少なくとも関係する環境を確認する。

- Realtime API 関連: `m5stack-core2-realtime`
- CoreS3 依存の変更: `m5stack-cores3-realtime`
- AtomS3R 依存の変更: `m5stack-atoms3r-realtime`
- Module LLM 関連: `m5stack-core2-llm` または `m5stack-cores3-llm`

依存ライブラリの取得などでネットワークが必要な場合は、ユーザーに確認してから実行する。

## ドキュメント方針

- 新しい仕組み、永続化データ、Web API、設定項目、タスクを追加した場合は `doc/fw_design.md` に追記する。
- 使い方が変わる場合は、README または該当する `doc/*.md` の更新も検討する。
- ドキュメントは長くしすぎず、後から実装の意図を追える情報を優先する。
